## Summary

Protocol health is strong for a four-feature backend foundation: every feature has a plan and a review, every review records DoD pass / approve, and the security documentation (threat model + SC1–SC16 constraints) is fully populated and internally consistent — no dangling or orphaned security IDs. The one real gap is a sequencing artifact of when the threat-model lifecycle landed: the four security-touching features (transport, crypto, identity, message-model) were all built and merged **before** `docs/threat-model.md` existed, so none of their plans links to the threat model or its SC-IDs, and none listed the threat model in "Docs to update" as the lifecycle now requires. Secondary findings are docs-currency lag (the module map still labels four shipped modules "Planned") and an unfinished code-review trail in all four review files.

## Notes

1. **Threat-model traceability gap on the four security features (the focus of this audit).** All four security-touching features — webdav-transport (transport/TLS), crypto (AEAD + key management), identity (signing, key agreement, sealed-box), message-model (per-message Ed25519 signing) — were merged (2026-06-03 / 2026-06-04) **before** `docs/threat-model.md` was authored (commit `61f26f7`, 2026-06-04 00:49). As a result, **none of their plans lists `docs/threat-model.md` in "Docs to update", and no plan or review references any `SCn` ID or Threat row** (verified: a grep for `threat-model` / `SCn` across all plans and reviews returns nothing). Under WORKFLOW.md "Threat-model lifecycle" + "Security-relevant surfaces", a feature touching cryptography / key-management / network-transport / authentication must now reference the threat model and its constraints. **Why it matters:** the link from a built security surface back to the threat it mitigates is the audit trail that lets the next security feature reason about what is already defended; without it, a future change to (say) the AEAD or the signature path can't quickly see which Threat rows it affects. **This is a sequencing artifact, not a protection gap** — see the "What looks healthy" coverage check below: the threat model authored afterward does cover all four surfaces. **Remediation:** spawn `pm-architect` to add a back-reference for each of the four features — either a one-line "Threats addressed: Tnn (SCmm)" annotation appended to each plan's contract/security section, or a coverage note in the threat model mapping each Threat to the feature that implemented its mitigation. This is doc reconciliation, not re-planning; no code changes.

2. **The threat model's `Last reviewed` predates nothing — but the lifecycle "per feature" step was never exercised for these four.** `docs/threat-model.md` `Last reviewed: 2026-06-04` is correct and not stale by date (it was authored after the last security feature, message-model, merged 2026-06-04 00:37; threat model committed 00:49). **Why it matters:** by the auditor's staleness rule this is *not* a stale-threat-model finding. It is recorded here only to close the focus question explicitly: the date is consistent, the document is populated, and the only outstanding item is the backward link in Note 1. No remediation beyond Note 1.

3. **`docs/architecture.md` "File layout (module map)" is stale — four shipped modules still labeled "Planned".** The map marks `app/.../crypto/`, `app/.../identity/`, `app/.../keystore/`, and `app/.../message/` as "Planned (feature not yet run)", but all four packages exist in the working tree with source files (e.g. `crypto/Aead.kt`, `identity/IdentityCrypto.kt`, `keystore/ChatKeyStore.kt`, `message/MessageParser.kt`). Only `transport/` and `protocol/` are correctly marked "Implemented". **Why it matters:** the module map is the agent-facing "where does code live" reference; a map that says a shipped module is unbuilt mis-routes the next feature's placement decision, and the map's own stated "honesty rule" (implemented rows are facts) is violated. **Remediation:** spawn `pm-architect` to reconcile the four rows to "Implemented" against the real tree.

4. **Code-review (Pass 2) trail is unfinished in all four review files.** Each `<topic>_review.md` ends with empty `## Code review findings` and an unstamped `## Code review` heading — none carries the `## Code review: <date> — passed` stamp the review loop is supposed to write when Pass 2 clears. **Why it matters:** the review file is the persistent record that both review passes ran; an unstamped Pass-2 section leaves it ambiguous whether technical-quality review happened and cleared, or was skipped. **Remediation:** for each feature, confirm `code-review` ran clean and have the orchestrator stamp the section `## Code review: <date> — passed` (or run `code-review` now if it never ran). This is the review-trail record, not a re-litigation of code quality.
