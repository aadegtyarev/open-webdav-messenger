## Quality sweep: 2026-06-06 — swept a92650c..HEAD at depth high

First-run full-tree smell/hygiene sweep (built-in `code-review` engine, high depth — the `wb-development:code-review-orchestrator` engine is not installed; no prior sweep marker existed). Scope: all 77 `.kt` files under `app/src/main`. Four finder angles (duplication+dead-code, complexity+length, crypto-security, transport/sync/codec) + adversarial verification. The complexity/length finder returned **nothing** — no AI-minimum violation (no function >50 logic lines, no nesting >2, no CC >10); the codebase is well-decomposed and KDoc-heavy. Verified findings below, grouped; all routed through PM triage.

### A. Duplication (directory ⟷ chatdirectory near-clones)
1. **Bounded-cursor reader triplicated** — `ChatDescriptorCodec.Cursor`, `DirectoryEntryCodec.Cursor`, and `message/ByteCursor` are the same bounds-checked `take/u8/u16(/u64)` reader (ByteCursor lacks `u64()`). Parse-security code on untrusted bytes; a bounds fix to one copy silently leaves the other two wrong. Extract one shared limit-bounded cursor. *(2 finders, CONFIRMED.)*
2. **`directory/` ≈ `chatdirectory/` packages** — `ChatDirectoryService`≈`DirectoryService`, `ChatDirectoryPaths`≈`DirectoryPaths`, `ChatDirectoryFactory`≈`DirectoryFactory`, `ChatDirectoryCrypto`≈`DirectoryCrypto`, differing only in the §10-vs-§11 collection name and grouping key. ~160-line orchestrators cloned; fixes must be made twice.
3. **Base32-lower alphabet re-declared 4×** — `MessageId.CONTENT_HASH_CHARS`, `ChangeEntry.CHAT_TAG_CHARS`, `DirectoryPaths.ENTRY_NAME_CHARS`, `ChatDirectoryPaths.ENTRY_NAME_CHARS` all spell the same RFC-4648 base32-lower set; `HashTag` already centralises `ORDER_TOKEN_CHARS` and should host this too.
4. **AEAD size constants duplicated** — `Aead.NONCE_BYTES=24`/`TAG_BYTES=16` and `ChatKey.KEY_BYTES=32` are hand-declared while `LazySodiumCrypto` independently validates against the libsodium `AEAD.XCHACHA20POLY1305_IETF_*` constants — two unsynchronised sources of truth. *(CONFIRMED.)*

### B. Dead code
5. **`ChatKey.destroy()` and `Identity.destroy()` — zero callers anywhere** (main/test/androidTest). Public best-effort-zeroize methods that nothing invokes: the secret material they target relies on GC while the surrounding code wipes transient copies — a false hygiene affordance. Wire into the drop-lifecycle or remove. *(CONFIRMED.)*
6. **`ChatKey.export()` ≡ `copyBytes()`** (`bytes.copyOf()`), zero main callers (test-only; the consuming invite/QR feature is deferred). Widens the raw-key escape surface for an unbuilt feature. Fold in or defer with the feature. *(CONFIRMED.)*
7. **`ChatPaths` gen-1 helpers** — `INBOX`, `inbox()`, 3-arg `message()` are test-only dead surface in a security-relevant path-minting object (live code uses the 2-arg `message()`; note `MessageId.inboxId` itself is still live via `changeIndex`). Move the strings into the legacy tests, delete from production. *(CONFIRMED.)*

### C. Crypto hygiene inconsistencies
8. **`LazySodiumCrypto` error-contract asymmetry** — `aeadEncrypt`/`argon2id`/`genericHash`/`keyedHash` **throw** `IllegalStateException` on libsodium failure, while `aeadDecrypt`/`boxSealOpen`/`signVerifyDetached` return null/false. `Aead.seal` and `MessageCrypto.sealEnvelope` have no catch, so an `aeadEncrypt` fault propagates **uncaught out of the send path** (decrypt-side faults degrade to typed `Rejected`). *(CONFIRMED — the one finding with a plausible user-visible effect: a crash on send under a spurious native failure.)*
9. **`Aead.open()` zeroization gap** — wipes only `key` in `finally`; the recovered plaintext and the copied `nonce` are left on the heap, inconsistent with the codebase's own wipe discipline (sibling buffers `key`/`derived`/`shared`/`serialized` are all zeroized). *(CONFIRMED.)*

### D. Architecture forethought (deferred-feature code — fix before the feature lands)
10. **`IdentityCrypto.agreeChatKey` has no chat-id binding** — the per-chat key is `keyedHash(CHATKEY_KDF_CONTEXT, shared, …)` with a **fixed** context constant; the only varying input is the X25519 DH `shared`, so **two distinct chats between the same identity pair derive the identical `ChatKey`**, weakening per-chat key isolation. **Zero production callers today** — it is scaffolding for the deferred directory / private-chat / rotation features. CONFIRMED. The right time to fix (mix the chat-id into the KDF input) is when those features are planned, before it ships. *(Also in this deferred cluster, lower: `importRawKey`/`openSealed` raw-overload don't wipe caller secrets — consistency to settle when the invite feature lands.)*

### E. Lower-severity notes (not separately triaged)
- `IdentityStore.generateOnce` `FileLock`↔`Mutex` coupling — correct today (the in-process Mutex funnels all entry) but fragile if a future caller bypasses it or a second `android:process` re-enters → `OverlappingFileLockException`. PLAUSIBLE, not a present defect. Worth a guard/comment.
- `KeystoreWrapper` forwards `e.message` into `UnwrapResult.reason` — benign today (Keystore/GCM messages carry no key bytes), but a forward-arbitrary-throwable-message channel by construction.
- `PropfindParser.getElementsByTagNameLocal` re-walks the full descendant list per local-name per response — a per-poll efficiency smell on the round-trip the design wants cheap.

**Cleared on inspection (considered, not missed):** untrusted-length parsing is bounded (`TlvFields.read` caps field-count, `WebDavTransport.readCapped` 1 MiB GET cap on real bytes, `BackOffPolicy` guards shl overflow), `CallExecutor.retryLoop` honours cancellation + bounded retries, path segments from untrusted listings are gated by well-formed-name alphabets (no traversal), no swallowed parse exceptions hide corruption (failures are typed `Rejected`/`NotReady`).
