# x25519-identity (remote private chats + D10 fix) — design notes

> Security-sensitive key-derivation change on a security-bearing project. Resolves the four structural/security choices the plan (`docs/features/x25519-identity_plan.md`) defers to the architect, BEFORE coding. This note guides the coder; it does not write code. Sources cited inline (libsodium docs via `docs/stack-notes.md` → Crypto "Public-key primitives"; `docs/architecture.md` decisions 9/10; `docs/threat-model.md` A2/T10/T15).

## Context

The plan is narrow: (1) fix blocker **D10** so a DH-derived chat key is **distinct per chat-id**, and (2) wire that DH-derived key into a usable backend path (directory-discovered peer + chat-id → derive → `ChatKeyStore.store`), **no secret exchanged over any channel**. The substrate is otherwise built: `IdentityCrypto.agreeChatKey` already does `crypto_box_beforenm` → keyed-BLAKE2b over a **fixed** context `owdm/x25519-chatkey/v1`, and `DirectoryService.readDirectory` already returns verified peers (`DirectoryEntry` carries `boxPublicKey`).

This qualifies for an arch note on both gate criteria: it **adds an axis of extension** (a chat-id-bound DH derivation alongside the existing pairwise primitive — a new domain-separation purpose), and there are **multiple plausible homes** for the production derivation + provisioning seam (`identity` / `directory` / `keystore` / a new coordinator).

## Adjacent implementations (the same kind of job, read in full)

1. **`IdentityCrypto.agreeChatKey`** at `identity/IdentityCrypto.kt:56` — DH → `keyedHash(CHATKEY_KDF_CONTEXT, shared, 32)`. The DH shared secret is the keyed-BLAKE2b **key**; the fixed context string is the **message**. Zeroizes `shared` and `derived` in `finally`. This is the exact shape D10 must extend — bind chat-id into the **message**.
2. **`KeySources.knownKey`** at `crypto/KeySources.kt:76` — `BLAKE2b(KNOWN_KEY_CONSTANT ‖ 0x1F ‖ chatId)`. **The in-house precedent for binding a chat-id into a hashed input.** It uses an explicit `0x1F` Unit-Separator domain separator between a **fixed-length-prefix-free constant and the variable-length chat-id** "so no two (constant, chatId) pairs can collide by concatenation." This is the canonicalization answer for choice 1's ambiguity question.
3. **`KeySources.saltForChat`** at `crypto/KeySources.kt:88` — `genericHash(SALT_CONTEXT ‖ chatId)` → deterministic Argon2id salt **derived from the chat-id, not random**. This is the in-house precedent for choice 1's salt question: a *public-input-derived* salt that both members reproduce, never a stored random salt.
4. **`DirectoryService.readDirectory` → `DirectoryEntry`** at `directory/DirectoryService.kt:85` / `directory/DirectoryEntry.kt` — the verified-peer output: `displayName`, `signingPublicKey`, `boxPublicKey` (32 bytes each), plus `toPublicIdentity()`. The provisioning seam consumes `DirectoryEntry.copyBoxPublicKey()`. **`DirectoryService` depends on `identity` (imports `Identity`), not the reverse** — the load-bearing dependency direction for choice 2.
5. **`ChatKeyStore.store(chatId, chatKey)`** at `keystore/ChatKeyStore.kt:31` — wraps + persists, **overwrites any existing key for that chatId** (atomic), file name = `BLAKE2b(chatId)` Base32 token. Source-agnostic: it neither knows nor cares whether the key came from passphrase/random/known/DH. Relevant to choice 4 (overwrite semantics).

## Behavioral risks in this area

Not event-driven — `deriveRemoteChatKey` is pure local crypto and `ChatKeyStore` is keyed by chat-id, so there is **no subscription/feedback loop** (the plan's "Interaction scenarios" confirm isolation). The risks are crypto-correctness and key-overwrite, addressed per choice below.

---

## Choice 1 — Chat-id binding in the KDF (the D10 fix)

**Chosen construction:** keep the existing two-stage shape — DH shared secret as the keyed-BLAKE2b **key**, a domain-separation string as the **message** — and **bind the chat-id into the message**, with an **explicit separator**, under a **bumped v2 context**:

```
shared   = crypto_box_beforenm(peerBoxPublic, myBoxSecret)            // DH output, the BLAKE2b KEY
chatKey  = keyedHash(  "owdm/x25519-chatkey/v2" ‖ 0x1F ‖ chatIdBytes,  // the BLAKE2b MESSAGE
                       key = shared, outLen = 32 )
```

- **Keyed-BLAKE2b message (not a `crypto_kdf` subkey-id, not a separate salt arg) is the right shape — rationale:** the DH output is correctly the BLAKE2b **key** (this is exactly the libsodium "do not feed the raw shared secret to the AEAD — run it through generichash/KDF" rule, stack-notes → Crypto "Public-key primitives", <https://doc.libsodium.org/public-key_cryptography/authenticated_encryption>). Keyed BLAKE2b's own definition mixes key + message, so a per-chat **message** yields a per-chat key with full domain separation — this is precisely how `KeySources.knownKey`/`saltForChat` already bind a chat-id. A `crypto_kdf` (`crypto_kdf_derive_from_key(subkey, id, ctx, masterKey)`) is **rejected**: it would force the chat-id into an 8-byte `subkey_id` + an 8-byte context, truncating/encoding the opaque variable-length chat-id into a fixed numeric slot — lossy and collision-prone for a chat-id whose grammar is still `[?]` (architecture §8). The keyed-hash message takes the chat-id's full bytes. This also keeps the construction **identical in kind** to the four other in-house derivations (one keyed/generic BLAKE2b call), minimizing the audited-surface footprint (no new primitive — `keyedHash` already exists in `NativeCrypto`).

- **Domain-separation / canonicalization — use a fixed-prefix context + an explicit separator, context FIRST:** `versioned-context ‖ 0x1F ‖ chatId`. The version string `owdm/x25519-chatkey/v2` is a **fixed, separator-free, non-extending prefix** and the chat-id is the **trailing variable-length field**, with an explicit `0x1F` Unit Separator between them — **byte-identical in spirit to `KeySources.knownKey` (`KNOWN_KEY_CONSTANT ‖ 0x1F ‖ chatId`, `crypto/KeySources.kt:77`).** This removes the concatenation-ambiguity concern: because the context never contains `0x1F` and is a fixed known string, there is exactly one parse of `ctx ‖ 0x1F ‖ chatId` — no `(ctxA, idA)` and `(ctxB, idB)` can produce the same byte string. Length-prefixing the chat-id is an acceptable alternative but is **not** the in-house idiom; match `knownKey`'s separator approach for consistency (one canonicalization convention across all four BLAKE2b derivations). Keep `context ‖ separator ‖ chatId` order (not `chatId ‖ context`) so the fixed, known prefix leads — same order as every existing context-prefixed derivation in the codebase.

- **No per-chat random salt — chat-id is sufficient AND a random salt is actively wrong here:** the load-bearing property is that **both members derive the SAME key from public inputs only** (plan Scenario 1, symmetry). A random salt would have to be agreed/shared between the two members — but there is **no secret-exchange channel** (that is the entire point of the feature: "no secret exchanged over any channel"). A shared random salt is itself a shared secret to distribute, defeating the no-secret-exchanged property. The chat-id is a **public, mutually-known coordinate** both members already hold, so it is the correct per-chat differentiator — exactly the reasoning `KeySources.saltForChat` (`crypto/KeySources.kt:88`) already applies for the Argon2id salt (deterministic-from-chat-id, never stored-random). The DH shared secret supplies all the keying entropy; BLAKE2b does not need an additional salt for uniformity.

- **v1 → v2 versioning is correct and safe — nothing on disk depends on v1.** Verified: `agreeChatKey` has **no production callers** (`grep` finds only its own KDoc mentions + tests in `identity/`). Nothing is persisted under the v1 context (a DH-derived key is only stored via the *new* provisioning seam this feature adds, which uses v2). So bumping the context to `owdm/x25519-chatkey/v2` for `deriveRemoteChatKey` is a pure additive change with **zero on-disk migration** — it marks the new chat-id-bound purpose explicitly (the stack-notes "distinct versioned context per derivation purpose" rule, <https://doc.libsodium.org/hashing/generic_hashing>), and leaves `agreeChatKey`'s v1 untouched for its existing tests.

## Choice 2 — Module homes for the production derivation + provisioning seam

**Chosen:** `deriveRemoteChatKey` lives in **`IdentityCrypto`** (alongside `agreeChatKey`); the **provisioning seam lives in a new thin coordinator in the `directory` package** (or, equivalently, a small `identity`-adjacent provisioner the wiring composes — see note), **never** by making `identity` depend on `directory`.

- **`deriveRemoteChatKey` → `IdentityCrypto`.** It is the same kind of job as `agreeChatKey` (DH → keyed-BLAKE2b → `ChatKey`), uses the same `NativeCrypto` primitives (`boxBeforeNm` + `keyedHash`), and shares the v2 context companion-constant home. Putting it anywhere else would split one cohesive crypto family across two homes and force a second module to reach for `boxBeforeNm`. It composes cleanly: `deriveRemoteChatKey` can be `agreeChatKey`'s body with the chat-id folded into the keyed-hash message (or both can call a shared private `derive(shared, contextMessage)` helper — coder's choice, behaviour-preserving for `agreeChatKey`). Keeps `identity` pure crypto (decision 10 — no on-disk format, no directory/keystore dependency).

- **Provisioning seam → a thin coordinator that depends "downward" on `identity` + `directory`'s output type + `keystore`.** The seam composes three already-existing modules: it takes the local `Identity` (its box secret), a peer's `boxPublic` (from a `DirectoryEntry`, the directory's verified-peer output), and a chat-id → calls `IdentityCrypto.deriveRemoteChatKey` → `ChatKeyStore.store`. **The dependency direction that must NOT be created is `identity → directory`** (identity is the lower, pure-crypto substrate; directory already depends on identity, per `DirectoryService.kt` importing `identity.Identity`). So the seam belongs **above** identity. Two acceptable homes, both keeping arrows sane:
  - **(Recommended) a new small provisioner type in the `directory` package** (e.g. `RemoteChatProvisioner` next to `DirectoryService`) — `directory` already legitimately depends on `identity` + `crypto`, and consuming `ChatKeyStore` (`keystore`) is a new, sane downward edge (directory → keystore; keystore depends on neither). This co-locates the seam with the `DirectoryEntry` it consumes and the "discover a peer" flow it extends.
  - **(Acceptable alternative) a free-standing tiny coordinator module** the app-wiring composes, depending on `identity` + `directory` + `keystore`. Cleaner layering if the team prefers the seam own no directory internals, but it adds a module for one function.
  - **Reject:** putting the seam in `identity` (would need `identity → directory`/`keystore`, inverting the substrate arrow) or in `keystore` (would need `keystore → identity`/`directory`, polluting the lowest storage layer). Exact name/home is the coder's choice (plan says so); the **binding constraint is the arrow direction**, not the name.

- **Typed result (C8 posture).** The seam returns a typed success/failure (plan Contracts) — a native DH/KDF failure degrades to a typed failure, **no uncaught throw**, exactly as `DirectoryService.publishEntry` maps a native seal `IllegalStateException` to `PublishOutcome.Failed` (decision 14 / T26). Catch the **narrow** native-crypto failure at the derivation boundary, not a blanket `catch (Throwable)` (decision 14 discipline — unrelated bugs must still surface).

## Choice 3 — Misuse-prevention typing for the bare `agreeChatKey`

**Chosen:** **KDoc-only mitigation for this slice + a named typed follow-up** — do **not** change `agreeChatKey`'s return type now.

- **Trade-off:** `agreeChatKey` returns a `ChatKey` that is NOT chat-id-bound — a real misuse hazard if a caller fed it straight to the AEAD. A compile-time guard (a distinct return type, e.g. `PairwiseSecret`, for the bare pairwise output) would be the stronger fix. **But** changing the return type **edits the existing `agreeChatKey` tests** (`IdentityCryptoTest`, `IdentityStackSpecTest`, `IdentityInstrumentedTest` all assert a `ChatKey`/its bytes), and the plan's hardest constraint is **zero existing-test edits** ("ALL existing tests, unchanged"). The misuse risk is **currently latent**: `grep` confirms `agreeChatKey` has **no production caller** — after this feature, production derives via `deriveRemoteChatKey` exclusively, so nothing reachable misuses the bare output today.
- **Therefore:** update `agreeChatKey`'s KDoc to mark it the **bare pairwise primitive, superseded-for-chat-keys by `deriveRemoteChatKey`, output must not be used directly as a chat key** (plan Contracts already calls for this), and record a **typed-distinct-return follow-up** (a `PairwiseSecret` wrapper that touches the bare-primitive tests) as a separate hardening — exactly the deferral the plan's "Out of scope" already names. A path that adds a *new* misuse-safe type **without** touching existing tests (e.g. a new `agreePairwiseSecret` returning the distinct type, leaving `agreeChatKey` as a deprecated alias) is possible but adds surface for a primitive with no caller — not worth it this slice. **Net:** accept KDoc-only now; the compile-time guard is the right *next* step, gated on a slice that is allowed to touch the bare-primitive tests.

## Choice 4 — Other security-relevant wiring observations

- **Key-overwrite / collision under one chat-id (the real one to flag).** `ChatKeyStore.store` **overwrites** any existing key for a chatId (`keystore/ChatKeyStore.kt:31`, "Overwrites any existing stored key for that chat"). So if a chat-id already has a passphrase/random/known key stored, provisioning a DH-derived key under the **same** chat-id **silently replaces it** — and the file name is `BLAKE2b(chatId)`, so the collision is by **chat-id value**, not by source. This is **not a flaw to fix in this slice** (the store is correctly source-agnostic; one chat-id = one key is the model), but the provisioning seam **must not be invoked with a chat-id that already names a different-source chat** — that is a caller (future chat-creation/invite UI) responsibility. **Coder action:** the seam's KDoc must state that `store` overwrites by chat-id, and the *idempotency* the plan relies on (re-provisioning the same peer+chat-id → same key → same logical result, plan Interaction scenario 2) holds **only because the derivation is deterministic** (same inputs → same key); the stored blob bytes differ per wrap (fresh Keystore IV), but the unwrapped key is identical. Do **not** add a "refuse if present" guard unless the plan asks — it does not.
- **Typed-failure-on-native-error path is mandatory, consistent with C8/T26.** The DH (`boxBeforeNm`) and KDF (`keyedHash`) are native calls that can fail; the provisioning seam must map that to its typed failure (plan: "no uncaught throw on the provisioning path"), narrow-catch only. This matches `DirectoryService`'s existing posture and threat-model **T26** (native seal failure → typed failure, no `SCn` weakened).
- **Zeroization parity (SC5 family / A2).** `deriveRemoteChatKey` must zeroize the DH `shared` and the derived intermediate in `finally`, exactly as `agreeChatKey` does (`IdentityCrypto.kt:60-72`) and as `DirectoryService.publishEntry` wipes its copied signing secret (`directory/DirectoryService.kt:69-72`). The box-secret copy the seam pulls from `Identity.copyBoxSecret()` must be wiped after the derive (the `openSealed(blob, identity)` overload at `IdentityCrypto.kt:101` is the in-house pattern: copy secret → use in `try` → `fill(0)` in `finally`). **A2 (identity secret keys)** never leaves memory; only the public `boxPublic` from the directory and the public chat-id are inputs alongside the local secret.
- **No new threat, no weakened `SCn` — a strengthening.** The D10 fix **closes** a per-chat key-isolation weakness (two chats between a pair shared one key → now distinct). The post-coding threat-model update (plan "Docs to update", owned by `pm-architect`) records DH-derived keys as per-chat in the A2/T-row family and confirms no SC weakened — flagged here so the coder knows the security direction is *tightening*, and the `deriveRemoteChatKey never equals agreeChatKey output` + `cross-chat isolation` tests are the regression evidence for it.

## Recommendation summary

- **Choice 1 — bind chat-id into the keyed-BLAKE2b *message* as `"owdm/x25519-chatkey/v2" ‖ 0x1F ‖ chatId`** (DH output stays the BLAKE2b key; explicit `0x1F` separator + fixed leading context = unambiguous, matching `KeySources.knownKey`); **no random salt** (would need a shared secret — defeats no-secret-exchanged; chat-id is the public differentiator, matching `saltForChat`); **v2 bump is safe** (`agreeChatKey` has no production caller, nothing persisted under v1).
- **Choice 2 — `deriveRemoteChatKey` in `IdentityCrypto`; provisioning seam in a thin `directory`-package coordinator** (or a free-standing coordinator) composing `identity` + `DirectoryEntry` + `ChatKeyStore` — **never `identity → directory`**; typed C8 failure result.
- **Choice 3 — KDoc-only mitigation now** (zero existing-test edits, no production caller misuses the bare output), **typed `PairwiseSecret` follow-up** deferred to a slice allowed to touch the bare-primitive tests.
- **Choice 4 — flag `ChatKeyStore.store` overwrites by chat-id** (seam must not collide a DH key onto a different-source chat-id; idempotency rests on deterministic derivation), **narrow typed-failure on native error** (T26 parity), **zeroize DH shared + box-secret copy** (SC5/A2). Net security direction: **tightening**, no `SCn` weakened.
