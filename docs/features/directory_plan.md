# directory — plan

> The **community user directory** on the shared WebDAV disk: each member publishes a signed, community-key-encrypted identity entry; members list + decrypt + verify the directory to discover peers and obtain their verified public keys. Backend/infrastructure — no UI yet (the chat surface, the QR safety-number screen, and the fingerprint display are later features). Consumes the done `identity` substrate (two public keys, detached sign/verify) and the done `crypto` substrate (AEAD seal/open) and the done `webdav-transport` layer. Defines a new on-disk section in `docs/protocol/webdav-layout.md`.
>
> **Scope decision (PM, 2026-06-04):** this feature is the **user/member directory only**. The sibling **chat directory** (a discoverable list of chats) is a separate later plan — see Out of scope.
>
> **Trust decision (PM, 2026-06-04):** entries are **self-published** — each member signs their own entry with their own identity key. Binding "this entry is really that person" is the **QR safety-number** verification (a later UI feature); this feature publishes verifiable, tamper-evident entries and the public-key material that verification consumes. Host-attested entries were considered and rejected for the MVP (see Out of scope).

## Scenarios

System behaviors (no UI yet — the directory engine consumers build on):

1. **Publish self.** A member publishes their identity entry to the community directory: a display-name plus their two **public** identity keys (Ed25519 signing, X25519 box — from the `identity` substrate), Ed25519-signed with their own signing key, then AEAD-encrypted with the **community key**, written to the disk as one entry per member. The community key, the signing, and the encryption are all reused primitives; this feature assembles them into an on-disk entry.
2. **Read + verify the directory.** A member lists the directory, AEAD-decrypts each entry with the community key, and verifies each entry's Ed25519 signature against the signing public key the entry carries. A valid entry yields `{display-name, signing-public-key, box-public-key}`. An entry that fails decryption (wrong/absent community key), fails signature verification, is malformed/truncated, or violates the path/name grammar is a **typed rejection** — dropped, never surfaced, never a crash — and the remaining valid entries still read.
3. **Discovery feeds downstream (not wired here).** The verified **X25519 box public key** of a peer is exactly what a future remote-private-chat (DH key agreement) needs; the pair of verified public keys is exactly what a future QR safety-number verifies. This feature **surfaces the verified entries** and the public-key material; it does **not** start chats, run key agreement, or render a fingerprint.
4. **Update own entry.** A member may re-publish their own entry (e.g. after reinstall, or a display-name change). The **newer** signed entry supersedes the older one for that member; a reader resolves to the latest valid entry per member (the supersede key + ordering are pinned in `webdav-layout.md` by this feature — a monotonic, signed, per-author version/timestamp, never a trusted clock for security).
5. **Unreadable without the community key.** A party with disk access but **without** the community key (the disk operator; an outsider who only obtained the app-password) sees **only ciphertext** — no names, no keys. The community key is supplied **out-of-band** (the future onboarding/invite feature distributes it; here it is supplied as configuration), is **never** the disk app-password, and is **never** written to the disk or logged.
6. **Robustness / flat-trust degradation.** A tampered, forged, garbage, wrong-key-encrypted, or truncated entry is a typed rejection (skipped, retried-next-cycle for a transient incomplete read — consistent with the §3 read discipline). Under the one shared credential any member can also **delete** another's entry; a member whose own entry was deleted simply re-publishes it. No single bad or missing entry wedges the directory read.

## Existing behaviors this feature touches

`docs/user-journeys.md` describes the send/receive journey (Journey 1) where membership and keys are arranged out-of-band. This feature is a first step toward in-app discovery but **does not change** that journey (no chat send/receive path is touched, no UI is added). Concrete touch points are code/contract:

- **Identity substrate** (`identity/`): consumes `publicIdentity()` (the two public keys) and `sign()` / `verify()` (Ed25519 detached, hard-reject on `-1`). Identity behavior, key storage, and the secret-key Keystore discipline must not change. Identity **secret** keys never enter a directory entry — only the public keys are published (SC5).
- **Crypto substrate** (`crypto/`): reuses the existing AEAD `seal` / `open` (XChaCha20-Poly1305) with the **community key** as the symmetric key — a community-wide symmetric key in the same family as the existing `random` / `known` sources (decision 9), not a per-chat key and not derived from the disk credential. The existing `ChatKey` type / AEAD must not change behavior. Whether the community key reuses the existing key-import/export path or gets a thin dedicated wrapper is an implementation detail for the architect/coder.
- **WebDAV transport** (`transport/`): reuses the existing verbs, `PROPFIND Depth: 1`, `MKCOL` (idempotent), append-only writes, content-hash-on-read, 429 back-off, and TLS-only (SC13). No new transport verb is needed.
- **On-disk protocol** (`docs/protocol/webdav-layout.md`): adds a new directory section (folder layout, entry framing, supersede/ordering, name grammar). Must not disturb §1–§9 (per-chat `log/` + `changes/`, message-id, envelope, message plaintext, cursor). The directory is **community-scoped**, distinct from the per-chat folders.

## Contracts

New, internal (names illustrative — the coder pins the exact API):

- **Publish** — `publishEntry(identity, displayName, communityKey) → PublishOutcome`: builds the entry (display-name + the two public keys), Ed25519-signs it with the identity signing key, AEAD-encrypts with the community key, writes it to the directory. Idempotent on re-publish of identical content; a changed entry is a new superseding version. Typed outcome, never throws.
- **Read** — `readDirectory(communityKey) → DirectoryReadResult`: lists the directory, decrypts + verifies each entry, returns the set of **verified** `DirectoryEntry { displayName, signingPublicKey, boxPublicKey }` (latest per member), plus a count/record of rejected entries (for diagnostics, not surfaced as members). Typed result, never throws.
- **DirectoryEntry** — the verified record: `{ displayName, signingPublicKey(32), boxPublicKey(32) }`. The public-key bytes are exactly the `identity` substrate's `publicIdentity()` shape, so a downstream DH / fingerprint consumer takes them directly.
- **(Local cache, if the architect places one)** — verified entries may be cached locally (Room) for offline browsing, analogous to local message history; the on-disk directory stays the source of truth. Whether a local cache lands in this feature or is deferred is an architect decision recorded in the arch note.

## Stack expectations touched

From `docs/stack-notes.md` (Crypto + OkHttp/WebDAV, Last reviewed 2026-06-03) and `docs/protocol/webdav-layout.md`:

- **Crypto — AEAD:** "XChaCha20-Poly1305 … 24-byte random nonce … `open` returns a typed rejection on any auth failure." The directory entry is sealed/opened with the **community key** via this same AEAD; the on-disk blob reuses the §5.1 framing (nonce ‖ AEAD-ciphertext-with-tag). Source: <https://doc.libsodium.org/secret-key_cryptography/aead> ; `docs/protocol/webdav-layout.md` §5.1.
- **Crypto — Ed25519 detached sign/verify:** "verify with `crypto_sign_verify_detached` (returns 0 on success, -1 on failure — treat -1 as a hard reject, never best-effort)." Every entry carries a 64-byte detached signature over its signed payload; verification failure is a hard reject (the entry is dropped). Source: <https://doc.libsodium.org/public-key_cryptography/public-key_signatures>.
- **Crypto — disk credential is never a content key (SC3 discipline):** the community key is "independent of disk access … never derived from the disk credential" — same rule as chat keys, extended to the community key. Source: `docs/architecture.md` SC3; `docs/stack-notes.md` Crypto.
- **OkHttp/WebDAV — `PROPFIND Depth: 1`, never infinity:** "Servers MUST support Depth 0/1; infinity MAY be disabled." The directory read is one `Depth: 1` listing of the directory collection. Source: <https://www.rfc-editor.org/rfc/rfc4918>; `docs/protocol/webdav-layout.md` §6.
- **OkHttp/WebDAV — `MKCOL` idempotent + 429 back-off + TLS:** the transport ensures the directory collection exists with `MKCOL` (already-exists handled idempotently); writes/reads tolerate 429 with exponential back-off; cleartext HTTP is rejected (SC13). Source: `docs/protocol/webdav-layout.md` §6; `docs/architecture.md` SC13.
- **On-disk integrity — append-only + content-addressing + on-read check:** "a writer only PUTs a new file, never overwrites; a reader rejects a file whose recomputed hash does not match its name." Directory entries follow the same append-only + content-addressed discipline as message files; an update is a new superseding file, not an in-place rewrite. Source: `docs/protocol/webdav-layout.md` §3.
- **On-disk — path-traversal rejection (SC16):** every minted/accepted directory path segment is restricted to the filename-safe alphabet (`[A-Za-z0-9._-~]`, no `/`, `..`, spaces); a value outside the grammar is rejected, never dereferenced. Source: `docs/architecture.md` SC16; `docs/protocol/webdav-layout.md` §0.

## Interaction scenarios

Not provably isolated — the directory shares the WebDAV disk, the one shared credential (flat trust), and the append-only / content-addressed discipline with the `sync` feature's per-chat folders, and it consumes the `identity` + `crypto` substrates.

- **Two members publish (or update) concurrently:** each entry is a new content-addressed file in the directory collection, so concurrent publishes never collide or overwrite — both land; a reader resolves the latest valid entry per member.
- **Publish while another member is reading:** a read in progress lists whatever entries are present at listing time; an entry written mid-read is simply picked up on the next read (no torn state — each entry is one self-contained file).
- **A member's entry is tampered or deleted (flat trust, SC11):** a tampered entry fails signature/AEAD verification and is dropped (the member appears absent until they re-publish); a deleted entry is gone until the owner re-publishes. No reader crash; the rest of the directory reads. The owner re-publishes to restore their entry.
- **An updated entry supersedes an older one:** a member who re-publishes (new display-name, or after reinstall) produces a newer signed version; a reader holding both resolves to the latest (per the signed supersede coordinate pinned in `webdav-layout.md`), and never shows a stale entry over a newer one or two entries for one member.
- **Directory coexists with per-chat folders on the same disk:** the directory collection is a distinct community-scoped location; reading/writing it does not touch any chat's `log/` or `changes/`, and the per-chat sync cycle does not touch the directory.
- **An entry is encrypted with the wrong community key (e.g. a stale key after a future rotation):** `open` fails as a typed rejection; the entry is dropped, not surfaced, not a crash.

## Test plan

- Existing tests that must pass: all `crypto`, `identity`, `webdav-transport`, `message-model`, and `sync` suites stay green.
- New tests (JVM unit — lazysodium-java + MockWebServer for the WebDAV layer):
  - `publish_then_read_roundtrips_verified_entry`: given an identity + community key, when an entry is published and then read back, then the verified `DirectoryEntry` carries the same display-name and the same two public keys.
  - `entry_is_ciphertext_only_on_disk`: when an entry is published, then the bytes written to the WebDAV disk contain neither the display-name nor the public keys in cleartext (only AEAD ciphertext + the public framing).
  - `read_without_community_key_yields_nothing_readable`: given an entry on disk, when read with a wrong/absent community key, then it is rejected (no `DirectoryEntry`), never a crash.
  - `tampered_entry_rejected`: given a published entry whose ciphertext or signed payload is flipped by one byte, when read, then it is dropped (AEAD/`verify` rejection), and other valid entries still read.
  - `wrong_signature_rejected`: given an entry whose signature does not match its carried signing key (or is signed by a different key), when read, then it is dropped (hard reject on `-1`).
  - `malformed_or_truncated_entry_rejected`: a garbage / truncated / wrong-magic entry is a typed rejection, not a bounds error or crash.
  - `updated_entry_supersedes_older`: given two valid entries for the same member (older + newer), when read, then exactly one entry (the newer) is returned for that member.
  - `multiple_members_listed`: entries from several members all read back as distinct verified `DirectoryEntry`s.
- Interaction scenario tests (one per Interaction scenario above):
  - `concurrent_publishes_both_land`: two members publish near-simultaneously; both entries are present and read back (no overwrite/collision).
  - `publish_during_read_picked_up_next_read`: an entry written after a read started is absent from that read and present on the next.
  - `deleted_entry_absent_until_republish`: deleting a member's entry removes them from the read; re-publishing restores them; no crash.
  - `tampered_entry_does_not_wedge_directory`: a single tampered/forged entry is dropped and the remaining valid entries still read (no wedge).
  - `directory_read_does_not_touch_chat_folders`: a directory read/write leaves a chat's `log/`/`changes/` untouched (and vice versa) — asserted against the protocol paths.
  - `wrong_key_entry_dropped_not_crash`: an entry sealed with a different community key is a typed rejection on read.
- Stack-spec tests (verify the cited rule; reference the source URL in a comment):
  - `entry_signature_hard_rejects_on_failure`: a bad/absent signature → `verify` false → entry dropped (libsodium `-1` hard reject — never accepted best-effort). Cites the public-key-signatures URL.
  - `entry_aead_uses_24_byte_random_nonce`: the on-disk entry blob carries a 24-byte nonce freshly generated per seal (reuses §5.1 framing). Cites the AEAD URL + `webdav-layout.md` §5.1.
  - `community_key_is_not_the_disk_credential`: the community key used to seal entries is independent of the WebDAV app-password and is never written to the disk (SC3 discipline). Cites SC3.
  - `directory_path_rejects_traversal`: a directory entry name containing `/`, `..`, or an out-of-alphabet character is rejected, never dereferenced (SC16). Cites SC16 + `webdav-layout.md` §0.
  - `propfind_depth_is_one`: the directory listing issues `PROPFIND Depth: 1` (not infinity). Cites RFC 4918 + `webdav-layout.md` §6.
- Instrumented tests (`connectedAndroidTest` — native + Keystore, decision 8 device gate):
  - `directory_native_seal_sign_roundtrip`: real lazysodium-android AEAD-seal + Ed25519-sign + verify of a directory entry on the device ABI with no `UnsatisfiedLinkError`.
  - `published_entry_uses_keystore_identity`: publishing uses the Keystore-wrapped identity signing key (secret key never leaves the keystore in plaintext); the published entry carries only the public key.

## Out of scope

- **Chat directory (sibling categorical element).** A discoverable list of *chats* (chat-id, kind, access, title) so a member finds joinable chats without a per-chat invite. Excluded because it is a distinct data shape with distinct trust questions (e.g. listing a private chat exposes its existence but joining still needs the key) — a separate plan. This feature is the **user/member** element of the directory only.
- **QR safety-number verification + fingerprint display (UI).** The act of verifying "this entry is really that person" and the human fingerprint format (digit count/grouping) are the UI feature. This feature publishes the verifiable entries and surfaces the public-key material the fingerprint is computed over (the `identity` substrate already yields the deterministic fingerprint bytes).
- **Onboarding / invite distribution of the community key.** How the community key reaches a joining member (the string + QR onboarding bundle) is a separate feature. Here the community key is supplied as configuration / out-of-band (the same deferral pattern as `sync`'s roster/connection config).
- **Remote-private-chat establishment (DH wiring).** Deciding to start a remote private chat, fetching a peer's published X25519 key, and running `agreeChatKey` into a real chat is a separate feature. This feature only **publishes and surfaces** the X25519 public key.
- **Host-attested entries / community ownership marker.** The host signing each member's entry (host-as-authority), and the signed `meta/community.json` owner marker, belong to the **community** feature. Rejected for this MVP per the PM trust decision (self-published + QR), recorded so the seam is understood: the directory's signed-entry format does not preclude a later host counter-signature.
- **Key rotation / member removal / individual revocation.** Re-keying the community or a chat, the rotate-with-auto-replace mechanism, and removing a member are separate future features (they consume the `identity` sealed-box + signing primitives). A removed/rotated member's stale-key entry simply fails to decrypt for others post-rotation (Scenario 6 wrong-key path); the rotation feature owns the active removal flow.
- **Roster file management (`meta/roster.json`).** Writing/signing the per-chat roster is a separate feature; the directory is community-scoped member discovery, not the per-chat member list the `sync` change-index notifies against.
- **Local-history-at-rest encryption** of any directory cache (if a cache lands) — same accepted limitation as SC17/T16 for message history; not introduced here.
