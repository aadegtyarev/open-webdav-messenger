# chat-directory — plan

Decision authority: autonomous

> The **community chat directory** on the shared WebDAV disk: the sibling of the done user directory (§10). Each member publishes a signed, community-key-encrypted **chat descriptor** (chat-id + kind + access + title); members list + decrypt + verify the chat directory to **discover joinable chats** without a per-chat invite, and resolve the latest valid descriptor per chat-id. Backend/infrastructure — no UI yet (the chat-list surface and the join action are later features). Consumes the done `identity` substrate (Ed25519 detached sign/verify), the done `crypto` substrate (community-key AEAD seal/open), and the done `webdav-transport` layer. Reuses the done `directory` (§10) framing patterns. Defines a new community-scoped section §11 in `docs/protocol/webdav-layout.md`; does **not** disturb §1–§10.
>
> **Scope decision (PM, 2026-06-04, escalated under autonomous mode — security-relevant fork):** the chat directory lists **public groups + private GROUPS** with `{chat-id, kind, access, title}`. A private group's **existence + title** are discoverable by onboarded community members (the "community barrier" tier — sealed under the community key, never visible to the disk operator), but a private chat's **content key is NEVER in the directory** — joining a private chat still requires the key out-of-band (the future invite/onboarding feature). **DMs (1:1 chats) are NEVER listed** — that would expose the community's social graph ("who talks to whom"); a `dm`-kind entry is a hard reject on read (privacy enforced at read, not only at publish). See Out of scope for the excluded siblings.
>
> **Trust decision (inherited from the directory feature, PM 2026-06-04):** chat descriptors are **self-published** — each member signs the descriptor they publish with their own Ed25519 identity key. The signature proves *who authored this descriptor version* (tamper-evidence on the bytes), **not** authority over the chat. Under the one shared credential (flat trust, SC11) any member can publish or supersede a descriptor for any chat-id; an **authoritative chat-ownership marker** (creator-signed / host-attested chat entries) is deferred — parallel to §10's deferred host-attestation.

## Scenarios

System behaviors (no UI yet — the chat-directory engine consumers build on). The scenario subjects ("a member") are illustrative; this feature ships **no user-visible surface** (discovery becomes a user journey at the UI feature) — backend substrate, the same framing as the §10 directory feature.

1. **Publish a chat descriptor.** A member publishes a chat's discovery descriptor — chat-id + kind (`group`) + access (`public`/`private`) + title — Ed25519-signed with their own identity signing key, then AEAD-encrypted with the **community key**, written to the disk as one content-addressed append-only entry file in the chat-directory collection. A `dm` kind is **rejected at publish** (never written) — DMs are not discoverable (PM scope decision). The community key, the signing, the encryption, and the content-addressing are all reused primitives; this feature assembles them into a chat-descriptor entry.
2. **Read + verify the chat directory.** A member lists the chat-directory collection, content-hash-checks then AEAD-decrypts each entry with the community key, and verifies each entry's Ed25519 signature against the signing public key the entry carries. A valid entry yields a verified `ChatDirectoryEntry {chat-id, kind, access, title, published-by}`. An entry that fails decryption (wrong/absent community key), fails signature verification, is malformed/truncated, carries a `dm` kind or an invalid kind/access combination, or violates the path/name grammar is a **typed rejection** — dropped, never surfaced, never a crash — and the remaining valid entries still read.
3. **Discovery feeds downstream (not wired here).** The verified `chat-id` + the `kind`/`access` (which imply the chat's key source — decision 9) are exactly what a future **join / onboarding** flow needs to request the chat key and join a discovered chat. This feature **surfaces the verified descriptors**; it does **not** join chats, fetch or derive any chat key, or render a chat list.
4. **Update a chat descriptor.** A member may re-publish a chat's descriptor (title change, access change, or after reinstall). The **newer** signed descriptor (higher signed `version-counter`) supersedes the older one **for that chat-id**; a reader resolves to the latest valid descriptor per chat-id (a signed, monotonic per-chat version-counter + content-hash tiebreak — never a trusted clock for security; the §10.5 pattern, grouped by chat-id instead of signing-pubkey).
5. **Unreadable without the community key.** A party with disk access but **without** the community key (the disk operator; an outsider who only obtained the app-password) sees **only ciphertext** — no chat-ids, no titles, no kinds. The community key is supplied **out-of-band** (the future onboarding/invite feature distributes it; here it is supplied as configuration), is **never** the disk app-password, and is **never** written to the disk or logged.
6. **DMs and private-chat keys are never exposed by the directory.** A `dm` (1:1) chat is **never** published or surfaced — no social-graph leak. A private **group**'s existence + title are discoverable by community members (community-barrier tier, the PM scope decision), but the private group's **content key is never in the directory**; a member who discovers a private group still needs the key out-of-band to read or join it.
7. **Robustness / flat-trust degradation.** A tampered, forged, garbage, wrong-key-encrypted, or truncated entry is a typed rejection (skipped; a transient incomplete read is retried next cycle — the §3/§10.6 read discipline). Under the one shared credential any member can **delete** another's entry or **publish a competing descriptor** for a chat-id; the latest signed version wins (the descriptor is advisory discovery metadata, not authoritative chat ownership). A member whose entry was deleted simply re-publishes it. No single bad or missing entry wedges the chat-directory read.

## Existing behaviors this feature touches

`docs/user-journeys.md` describes the send/receive journey (Journey 1) where chats are arranged out-of-band. This feature is a further step toward in-app discovery but **does not change** that journey (no chat send/receive path is touched, no UI is added). Concrete touch points are code/contract:

- **Directory substrate** (`directory/`, §10): the chat directory is a **second** community-scoped collection alongside `directory/`. It reuses the §10 patterns — the §5 envelope frame, community-key AEAD seal/open, per-entry Ed25519 detached sign/verify (hard-reject on `-1`), content-addressed append-only entry names, the signed monotonic supersede coordinate, and the Depth-1 reject-don't-guess read — but with a **distinct inner payload** (a chat descriptor, not a member identity) and a **distinct collection**. The §10 user-directory behavior, its on-disk format, and its API must **not** change.
- **Identity substrate** (`identity/`): consumes `sign()` / `verify()` (Ed25519 detached, hard-reject on `-1`) and the author's `signing` public key (published to attribute the descriptor version). Identity secret keys never enter a chat-directory entry and never leave the Keystore (SC5); identity behavior and key storage must not change.
- **Crypto substrate** (`crypto/`): reuses the existing AEAD `seal` / `open` (XChaCha20-Poly1305) with the **community key** as the symmetric key — the same fourth-family community-wide symmetric key the §10 directory uses (decision 9 family), not a per-chat key and not derived from the disk credential. The `ChatKey` type / AEAD must not change behavior.
- **WebDAV transport** (`transport/`): reuses the existing verbs, `PROPFIND Depth: 1`, `MKCOL` (idempotent), append-only writes, `readContentAddressed` (content-hash-on-read), 429 back-off, and TLS-only (SC13). No new transport verb is needed.
- **On-disk protocol** (`docs/protocol/webdav-layout.md`): adds a new community-scoped section **§11** (chat-directory collection location, entry framing, inner chat-descriptor TLV, supersede/ordering, name grammar, read discipline). Must not disturb §1–§10 (per-chat layout, message-id, envelope, message plaintext, cursor, **and the §10 user directory**). The chat directory is a distinct community-scoped collection.

## Contracts

New, internal (names illustrative — the coder pins the exact API):

- **Publish** — `publishChatEntry(identity, chatId, kind, access, title, versionCounter, communityKey) → PublishOutcome`: builds the chat descriptor (chat-id + kind + access + title), Ed25519-signs it with the identity signing key, AEAD-encrypts with the community key, writes it to the chat-directory collection. **Rejects a `dm` kind** with a typed outcome (never writes a DM). Idempotent on re-publish of identical content; a changed descriptor is a new superseding version. Typed outcome, never throws.
- **Read** — `readChatDirectory(communityKey) → ChatDirectoryReadResult`: lists the chat-directory collection (one `PROPFIND Depth: 1`), content-hash-checks + decrypts + verifies each entry, **drops `dm`-kind and invalid kind/access entries**, and returns the set of **verified** `ChatDirectoryEntry` (latest per chat-id), plus a count of rejected entries (diagnostics, not surfaced) and a `listingFailed` flag (transport back-off → retry next cycle). Typed result, never throws.
- **ChatDirectoryEntry** — the verified record: `{ chatId, kind (group), access (public|private), title, publishedBySigningKey(32) }`. `chatId` is carried opaque/length-prefixed (its grammar is still `[?]` in §8 — pinned by a later feature, same deferral as `message-model`). `publishedBySigningKey` is the 32-byte Ed25519 key that signed this descriptor version (who authored it — not chat-ownership authority).
- **Kind / Access** — `kind ∈ {dm, group}` and `access ∈ {public, private}` per the architecture chat taxonomy (two-axis kind × access). In the chat directory **only `group` is valid** (`dm` is rejected at publish and on read); `group` may be `public` or `private`. An `access` outside `{public, private}` or a `dm`-kind entry is a typed rejection (reject-don't-guess).

## Stack expectations touched

From `docs/stack-notes.md` (Crypto + OkHttp/WebDAV, Last reviewed 2026-06-03) and `docs/protocol/webdav-layout.md` §10 (the §11 sibling reuses the same rules):

- **Crypto — AEAD:** "XChaCha20-Poly1305 … 24-byte random nonce … `open` returns a typed rejection on any auth failure." The chat-descriptor entry is sealed/opened with the **community key** via this same AEAD; the on-disk blob reuses the §5.1 framing (nonce ‖ AEAD-ciphertext-with-tag, header as AAD). Source: <https://doc.libsodium.org/secret-key_cryptography/aead> ; `docs/protocol/webdav-layout.md` §5.1 / §10.2.
- **Crypto — Ed25519 detached sign/verify:** "verify with `crypto_sign_verify_detached` (returns 0 on success, -1 on failure — treat -1 as a hard reject, never best-effort)." Every entry carries a 64-byte detached signature over its signed payload; verification failure is a hard reject (the entry is dropped). Source: <https://doc.libsodium.org/public-key_cryptography/public-key_signatures>.
- **Crypto — disk credential is never a content key (SC3 discipline):** the community key is "independent of disk access … never derived from the disk credential" — same rule as chat keys and the §10 directory, extended to the chat directory. Source: `docs/architecture.md` SC3/SC19; `docs/stack-notes.md` Crypto.
- **OkHttp/WebDAV — `PROPFIND Depth: 1`, never infinity:** "Servers MUST support Depth 0/1; infinity MAY be disabled." The chat-directory read is one `Depth: 1` listing of the chat-directory collection. Source: <https://www.rfc-editor.org/rfc/rfc4918>; `docs/protocol/webdav-layout.md` §6 / §10.6.
- **OkHttp/WebDAV — `MKCOL` idempotent + 429 back-off + TLS:** the transport ensures the chat-directory collection exists with `MKCOL` (already-exists handled idempotently); writes/reads tolerate 429 with exponential back-off; cleartext HTTP is rejected (SC13). Source: `docs/protocol/webdav-layout.md` §6; `docs/architecture.md` SC13.
- **On-disk integrity — append-only + content-addressing + on-read check:** "a writer only PUTs a new file, never overwrites; a reader rejects a file whose recomputed hash does not match its name." Chat-directory entries follow the same append-only + content-addressed discipline; an update is a new superseding file, not an in-place rewrite. Source: `docs/protocol/webdav-layout.md` §3 / §10.4.
- **On-disk — path-traversal rejection (SC16):** every minted/accepted entry name is restricted to the filename-safe Base32-lower alphabet (`[a-z2-7]`, no `/`, `..`, spaces); a name outside the grammar is rejected, never dereferenced. Source: `docs/architecture.md` SC16; `docs/protocol/webdav-layout.md` §0 / §10.4.

## Interaction scenarios

Not provably isolated — the chat directory shares the WebDAV disk, the one shared credential (flat trust), the community-root, and the append-only / content-addressed discipline with the `sync` per-chat folders and the §10 user `directory/`, and it consumes the `identity` + `crypto` substrates.

- **Two members publish (or update) the same chat's descriptor concurrently:** each entry is a new content-addressed file, so concurrent publishes never collide or overwrite — both land; a reader resolves the latest valid descriptor per chat-id (highest signed version-counter, content-hash tiebreak).
- **Publish while another member is reading:** a read in progress lists whatever entries are present at listing time; an entry written mid-read is picked up on the next read (no torn state — each entry is one self-contained file).
- **A descriptor is tampered or deleted (flat trust, SC11):** a tampered entry fails signature/AEAD verification and is dropped (the chat appears absent until re-published); a deleted entry is gone until re-published. No reader crash; the rest of the directory reads.
- **A `dm`-kind entry is forged into the collection:** a reader **drops** it (privacy enforced at read — DMs are never surfaced), and the remaining valid group entries still read.
- **An updated descriptor supersedes an older one:** a member re-publishes (new title) → a newer signed version; a reader holding both resolves to the latest per chat-id, never showing a stale descriptor over a newer one or two descriptors for one chat-id.
- **Chat directory coexists with the user directory and per-chat folders on the same disk:** the chat-directory collection is a distinct community-scoped location; reading/writing it touches **neither** the §10 `directory/` **nor** any chat's `log/`/`changes/`/`meta/`, and the per-chat sync cycle and the user-directory read do not touch the chat-directory collection.
- **An entry is encrypted with the wrong community key (e.g. a stale key after a future rotation):** `open` fails as a typed rejection; the entry is dropped, not surfaced, not a crash.

## Test plan

- Existing tests that must pass: all `crypto`, `identity`, `webdav-transport`, `message-model`, `sync`, and **`directory` (§10)** suites stay green (the §10 user directory must be byte-for-byte unaffected).
- New tests (JVM unit — lazysodium-java + MockWebServer / fake disk for the WebDAV layer):
  - `publish_then_read_roundtrips_verified_chat_entry`: given an identity + community key, when a group chat descriptor is published and read back, then the verified `ChatDirectoryEntry` carries the same chat-id, kind, access, and title.
  - `chat_entry_is_ciphertext_only_on_disk`: when an entry is published, then the bytes written to the disk contain neither the chat-id, the title, nor the kind/access in cleartext (only AEAD ciphertext + the public §5 framing).
  - `read_without_community_key_yields_nothing_readable`: given an entry on disk, when read with a wrong/absent community key, then it is rejected (no `ChatDirectoryEntry`), never a crash.
  - `tampered_chat_entry_rejected`: given a published entry whose ciphertext or signed payload is flipped by one byte, when read, then it is dropped (AEAD/`verify` rejection), and other valid entries still read.
  - `wrong_signature_rejected`: given an entry whose signature does not match its carried signing key, when read, then it is dropped (hard reject on `-1`).
  - `malformed_or_truncated_chat_entry_rejected`: a garbage / truncated / wrong-magic / wrong-version entry is a typed rejection, not a bounds error or crash.
  - `dm_kind_rejected_on_publish`: when `publishChatEntry` is called with `kind = dm`, then it returns a typed rejection and **writes nothing** to the disk (DMs are not discoverable).
  - `dm_kind_entry_dropped_on_read`: given a forged `dm`-kind entry on disk (validly sealed/signed), when read, then it is dropped and never surfaced as a `ChatDirectoryEntry` (privacy enforced at read).
  - `invalid_kind_access_combo_rejected`: an entry with an `access` outside `{public, private}` (or any non-`group` kind) is a typed rejection on read.
  - `updated_chat_entry_supersedes_older`: given two valid descriptors for the same chat-id (older + newer version-counter), when read, then exactly one (the newer) is returned for that chat-id.
  - `multiple_chats_listed`: descriptors for several distinct chat-ids all read back as distinct verified `ChatDirectoryEntry`s.
  - `private_group_listed_without_key`: a private-group descriptor reads back with `access = private` and its title, and the `ChatDirectoryEntry` carries **no** content key (the directory surfaces existence + metadata, never the chat key).
- Interaction scenario tests (one per Interaction scenario above):
  - `concurrent_publishes_same_chat_both_land`: two members publish a descriptor for the same chat-id near-simultaneously; both entries are present and the latest version-counter is resolved (no overwrite/collision).
  - `publish_during_read_picked_up_next_read`: an entry written after a read started is absent from that read and present on the next.
  - `tampered_chat_entry_does_not_wedge_directory`: a single tampered/forged entry is dropped and the remaining valid entries still read (no wedge).
  - `forged_dm_entry_dropped_others_read`: a forged `dm`-kind entry is dropped while the remaining valid group entries still read.
  - `updated_descriptor_resolves_latest_per_chat_id`: older + newer descriptor for one chat-id → exactly the newer is surfaced.
  - `chat_directory_read_does_not_touch_user_directory_or_chat_folders`: a chat-directory read/write leaves `directory/` and a chat's `log/`/`changes/` untouched (and vice versa) — asserted against the protocol paths.
  - `wrong_key_chat_entry_dropped_not_crash`: an entry sealed with a different community key is a typed rejection on read.
- Stack-spec tests (verify the cited rule; reference the source URL in a comment):
  - `chat_entry_signature_hard_rejects_on_failure`: a bad/absent signature → `verify` false → entry dropped (libsodium `-1` hard reject — never best-effort). Cites the public-key-signatures URL.
  - `chat_entry_aead_uses_24_byte_random_nonce`: the on-disk entry blob carries a 24-byte nonce freshly generated per seal (reuses §5.1 framing). Cites the AEAD URL + `webdav-layout.md` §5.1.
  - `community_key_is_not_the_disk_credential`: the community key used to seal chat entries is independent of the WebDAV app-password and is never written to the disk (SC3/SC19 discipline). Cites SC3/SC19.
  - `chat_directory_path_rejects_traversal`: an entry name containing `/`, `..`, or an out-of-alphabet character is rejected, never dereferenced (SC16). Cites SC16 + `webdav-layout.md` §0/§10.4.
  - `chat_directory_propfind_depth_is_one`: the chat-directory listing issues `PROPFIND Depth: 1` (not infinity). Cites RFC 4918 + `webdav-layout.md` §6/§10.6.
- Instrumented tests (`connectedAndroidTest` — native + Keystore, decision 8 device gate):
  - `chat_directory_native_seal_sign_roundtrip`: real lazysodium-android AEAD-seal + Ed25519-sign + verify of a chat-descriptor entry on the device ABI with no `UnsatisfiedLinkError`.
  - `published_chat_entry_uses_keystore_identity`: publishing uses the Keystore-wrapped identity signing key (secret key never leaves the keystore in plaintext); the published entry carries only the public key.

## Out of scope

- **DMs (1:1 chats) in the directory (sibling categorical element, PM scope decision 2026-06-04).** A `dm` is `kind = dm` in the two-axis taxonomy; it is **never** listed or surfaced — listing DMs would expose the community's social graph ("who talks to whom"). Excluded by an explicit, escalated PM decision; enforced as a hard reject at publish and on read. A DM is reached only via its own chat-join invite (the onboarding feature), never via discovery.
- **Private-chat KEY distribution.** How a member who *discovers* a private group obtains its content key (the string + QR chat-join invite) is the **onboarding/invite** feature. Here the directory surfaces existence + metadata only; the key is never in the directory.
- **Join / onboarding flow.** Deciding to join a discovered chat, requesting/receiving its key, and wiring it into a real chat is a separate feature. This feature only **publishes and surfaces** chat descriptors.
- **Authoritative chat-ownership marker / host-attested chat entries.** A creator-signed or host-counter-signed "this descriptor is authoritative for this chat" marker (so a member cannot spoof another chat's title/access) belongs to the **community / rotation** features. Deferred for the MVP per the inherited flat-trust trust decision (self-published + signed-for-tamper-evidence), recorded so the seam is understood: the signed-descriptor format does not preclude a later authority counter-signature. Parallel to §10's deferred host-attestation.
- **`chat-id` grammar pinning.** The chat-id is carried opaque/length-prefixed here (its grammar is still `[?]` in §8 `docs/architecture.md` / `webdav-layout.md`); pinning the grammar is a later feature, the same deferral pattern as `message-model`.
- **UI (chat-list surface, join button, public-chat warning).** Rendering the discovered chats, the join action, and the "public chat is not secret" warning are the UI feature.
- **Local Room cache of chat-directory entries.** Verified descriptors are recomputed from the on-disk source of truth per read; the Room cache + observable `Flow` for offline browsing is the **UI feature's** to add (the same deferral as §10, arch Option 4A). Local-history-at-rest encryption of any such cache is the same accepted limitation as SC17/T16.
- **Per-member revocation / re-key of directory access.** Removing one member's read access requires re-keying the community — the future key-rotation feature (same class as the §10/T21 community-key bearer-secret limitation).
- **Sibling: the user directory (§10).** The member/identity directory is the **done** `directory` feature; this plan is the **chat** element only. The two are distinct community-scoped collections with distinct inner payloads and distinct trust questions.
